# JBS Offline License Suite

This repository provides two Python applications:

* **JBS client** (`jbs_client/`) – an offline desktop application that verifies signed licenses and manages activation/deactivation files.
* **JBS License Generator** (`generator/`) – a GUI tool for producing license bundles, managing activations and issuing signed responses.

Both applications use RSA (3072-bit) signatures for integrity and AES-256-GCM for local storage encryption. The generator holds the RSA private key, while the client ships with the matching public key only.

## Python environment

```bash
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

## Running the applications

```bash
# Run the client (expects license.jbslic next to the executable or in %APPDATA%/JBS/)
python -m jbs_client.main

# Run the generator GUI
python -m generator.main
```

## PyInstaller builds (Windows 10/11)

Create the PyInstaller executables from an activated virtual environment:

```bash
pyinstaller jbs_client.spec
pyinstaller generator.spec
```

The client spec bundles the localisation JSON files and the embedded `public.pem`. The generator spec produces `JBS_Generator.exe`.

## Testing

Pytest covers the cryptographic primitives, HWID calculation and the activation lifecycle:

```bash
pytest
```

## Key management

* Use `generator/gui/main.py` to generate a fresh RSA key pair or load an existing `private.pem`.
* Store the private key securely (consider converting it to the encrypted JSON container using `jbs_common.crypto.encrypt_private_key`).
* Distribute only the `public.pem` with the client build.

## Offline activation workflow

1. The end-user installs a package generated by the generator GUI (`JBS.exe`, `license.jbslic`, `README.txt`).
2. On first launch the client produces a `.jbsreq` activation request containing the hardware fingerprint (HWID).
3. The supplier imports the request in the generator, verifies the license still has free slots and issues a signed `.jbsact` activation file.
4. The client imports `.jbsact`, validates the RSA signature and unlocks the application.
5. Deactivation follows an analogous flow via `.jbsunreq`/`.jbsunact`.

The local state files (`activations.dat`, `state.dat`) are encrypted with keys derived from the license identifier using Argon2id + AES-256-GCM.
